## 1. 需求分析

### 1.1 云同步功能

#### 核心需求
- **多设备同步**：用户可在手机、平板等多设备间无缝切换，数据实时一致
- **离线优先**：无网络时仍可完整使用，联网后自动同步变更
- **冲突解决**：当多设备同时修改同一数据时，需有明确的冲突处理策略

#### 同步机制
| 同步类型 | 触发条件 | 说明 |
|----------|----------|------|
| 实时同步 | 网络可用 + 数据变更 | 自动后台同步，用户无感知 |
| 手动同步 | 用户下拉刷新 | 强制拉取最新数据 |
| 启动同步 | App 启动时 | 检查并同步离线期间的变更 |

#### 同步策略
- **增量同步**：仅同步变更数据，节省流量
- **批量合并**：短时间内多次操作合并为一次同步请求
- **版本控制**：每条记录带版本号/时间戳，用于冲突检测

---

### 1.2 分享功能

#### 用户如何分享列表？
| 分享方式 | 说明 | 使用场景 |
|----------|------|----------|
| 公开链接 | 生成唯一 URL，任何人可访问 | 社交媒体分享、公开推荐 |
| 指定人分享 | 通过账号/邮箱邀请特定用户 | 私密协作、朋友间分享 |
| 二维码 | 生成二维码，扫码访问 | 线下场景、快速分享 |
| 导出分享 | 导出为图片/文本/JSON | 跨平台分享、备份 |

#### 权限控制模型
```
权限级别：
├── 私有 (Private)      - 仅自己可见
├── 指定人可见 (Shared) - 仅被邀请用户可见
└── 公开 (Public)       - 任何人可通过链接访问

操作权限：
├── 查看 (View)         - 只能浏览列表
├── 评论 (Comment)      - 可添加评论/备注
└── 编辑 (Edit)         - 可修改列表内容（协作者）
```

#### Fork 机制
- 任何公开/可访问的列表都可 Fork
- Fork 后生成独立副本，与原列表无关联（默认）
- 可选"关联原列表"：接收原列表更新通知

---

### 1.3 技术约束

- **后端方案**：Firebase（Authentication + Firestore + Storage）
- **匿名登录**：降低使用门槛，支持后续绑定邮箱
- **数据模型**：保持 Phase 1 的 Room 结构，增加同步元数据
- **安全性**：分享链接带 token 验证，防止未授权访问

## 2. 用户故事

### US-01: 多设备同步
```
作为 多设备用户，
我希望 我的列表在手机和平板间自动同步，
以便 在任何设备上都能访问最新数据
```

**验收标准：**
1. 在设备 A 创建/修改列表后，设备 B 在 5 秒内（网络正常）可见变更
2. 无网络时操作的数据，联网后自动同步到云端和其他设备
3. 同步过程中 UI 显示同步状态（同步中/已同步/同步失败）

---

### US-02: 匿名登录与账号绑定
```
作为 新用户，
我希望 无需注册即可开始使用，
以便 降低使用门槛，后续再决定是否绑定账号
```

**验收标准：**
1. 首次启动 App 自动创建匿名账号，无需任何操作
2. 匿名用户可使用全部核心功能（创建列表、同步、分享）
3. 支持将匿名账号绑定到邮箱/Google 账号，绑定后数据不丢失

---

### US-03: 生成分享链接
```
作为 列表所有者，
我希望 一键生成分享链接，
以便 通过微信/微博等渠道分享给朋友
```

**验收标准：**
1. 在列表详情页点击"分享"按钮，生成唯一 URL 链接
2. 链接可直接复制到剪贴板，或通过系统分享菜单发送
3. 链接带访问权限标识（公开/指定人），接收方根据权限访问

---

### US-04: 指定人协作
```
作为 项目组织者，
我希望 邀请特定用户协作编辑列表，
以便 和朋友一起维护旅行清单或活动计划
```

**验收标准：**
1. 可通过账号/邮箱邀请用户成为协作者
2. 协作者收到通知，接受后获得编辑权限
3. 列表所有者可随时移除协作者或更改权限

---

### US-05: Fork 公开列表
```
作为 内容消费者，
我希望 Fork 别人分享的公开列表，
以便 基于他人的推荐清单创建自己的版本
```

**验收标准：**
1. 访问公开链接时，显示"Fork 到我的列表"按钮
2. Fork 后生成完全独立的副本，可自由修改不影响原列表
3. Fork 时可选"关联原列表"，接收原列表更新通知

---

### US-06: 同步冲突处理
```
作为 多设备用户，
我希望 系统自动处理同步冲突，
以便 不用担心数据丢失或覆盖
```

**验收标准：**
1. 当检测到冲突时（同一记录多设备同时修改），自动采用"最后写入优先"策略
2. 被覆盖的变更保存到版本历史，用户可查看/恢复
3. 重大冲突（如删除 vs 修改）弹出提示让用户手动选择

---

### US-07: 分享权限管理
```
作为 列表所有者，
我希望 管理分享权限，
以便 控制谁能访问我的列表
```

**验收标准：**
1. 可随时切换列表的公开状态（私有↔指定人↔公开）
2. 公开链接可设置有效期（7 天/30 天/永久）
3. 可查看访问记录（谁在何时访问过）

---

### US-08: 离线模式
```
作为 经常出差的用户，
我希望 无网络时也能完整使用 App，
以便 在飞机/地铁上随时记录
```

**验收标准：**
1. 无网络时所有核心功能（创建/编辑/删除列表）正常使用
2. 离线操作的数据本地存储，联网后自动同步
3. UI 清晰显示当前网络状态（在线/离线/同步中）

## 3. 优先级排序（MoSCoW）

### Must Have（必须有）- Phase 2 核心
| 用户故事 | 优先级 | 理由 |
|----------|--------|------|
| US-01 多设备同步 | Must | 云同步的核心价值 |
| US-02 匿名登录与账号绑定 | Must | 用户系统基础 |
| US-06 同步冲突处理 | Must | 数据完整性保障 |
| US-08 离线模式 | Must | 产品核心定位（离线优先） |

### Should Have（应该有）- Phase 2 重要功能
| 用户故事 | 优先级 | 理由 |
|----------|--------|------|
| US-03 生成分享链接 | Should | 分享功能的基础 |
| US-05 Fork 公开列表 | Should | 差异化功能（Git 式分享） |
| US-07 分享权限管理 | Should | 隐私控制必要功能 |

### Could Have（可以有）- Phase 2 可选/Phase 3
| 用户故事 | 优先级 | 理由 |
|----------|--------|------|
| US-04 指定人协作 | Could | 高级协作场景，可延后 |

### Won't Have（本次不做）
| 功能 | 理由 |
|------|------|
| 评论功能 | 增加复杂度，Phase 3 考虑 |
| 版本历史 UI | 后台实现即可，用户界面可延后 |
| 访问统计 | 非核心需求 |
| Web 版分享页面 | 优先保证 App 体验 |

## 4. 数据模型扩展

### 4.1 新增 Entity

```kotlin
// 用户账号
data class UserAccount(
    val uid: String,           // Firebase UID
    val isAnonymous: Boolean,  // 是否匿名账号
    val email: String?,        // 绑定邮箱
    val createdAt: Long,
    val lastSyncAt: Long?
)

// 同步元数据
data class SyncMetadata(
    val localVersion: Long,     // 本地版本号
    val remoteVersion: Long,    // 云端版本号
    val lastModified: Long,     // 最后修改时间
    val isDeleted: Boolean,     // 软删除标记
    val conflictResolved: Boolean  // 冲突是否已解决
)

// 分享配置
data class ShareConfig(
    val listId: String,
    val visibility: Visibility,      // PRIVATE, SHARED, PUBLIC
    val shareToken: String?,         // 公开链接 token
    val expiresAt: Long?,            // 链接过期时间
    val collaborators: List<Collaborator>  // 协作者列表
)

data class Collaborator(
    val userId: String,
    val permission: Permission,  // VIEW, COMMENT, EDIT
    val invitedAt: Long,
    val acceptedAt: Long?
)

enum class Visibility { PRIVATE, SHARED, PUBLIC }
enum class Permission { VIEW, COMMENT, EDIT }
```

### 4.2 Firestore 数据结构

```
users/{uid}
├── profile: { isAnonymous, email, createdAt }
└── settings: { theme, notifications }

lists/{listId}
├── owner: { uid, name }
├── templateId: string
├── title: string
├── visibility: string
├── shareToken: string
├── items: subcollection
│   └── {itemId}: { name, status, fields, version, ... }
└── collaborators: subcollection
    └── {userId}: { permission, invitedAt, ... }

forks/{forkId}
├── originalListId: string
├── forkedFrom: { uid, listId }
└── isLinked: boolean
```

## 5. API 设计（概要）

### 5.1 同步接口

```
POST /sync/push      - 上传本地变更
GET  /sync/pull      - 拉取云端变更
POST /sync/conflict  - 上报/解决冲突
```

### 5.2 分享接口

```
POST /lists/{id}/share       - 生成分享链接
DELETE /lists/{id}/share     - 撤销分享
POST /lists/{id}/collaborate - 邀请协作者
GET  /share/{token}          - 通过 token 访问列表
POST /lists/{id}/fork        - Fork 列表
```

## 6. 风险与应对

| 风险 | 影响 | 应对方案 |
|------|------|----------|
| Firebase 国内访问不稳定 | 高 | 准备 Supabase 备选方案 |
| 同步冲突频繁 | 中 | 优化冲突检测算法，增加版本历史 |
| 分享链接被滥用 | 中 | token 加密 + 访问频率限制 |
| 匿名用户数据丢失 | 高 | 强提醒用户绑定账号，提供导出备份 |
