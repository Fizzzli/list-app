## 开发流程（自迭代）

```
┌─────────────────────────────────────────────────────────┐
│                    Phase 开始                            │
└─────────────────────────────────────────────────────────┘
                           │
                           ▼
┌─────────────────────────────────────────────────────────┐
│  1. PM 拆解任务 → 创建 GitHub Issues                      │
│     - 每个 Issue 是一个可独立交付的功能                    │
│     - 标注优先级 (P0/P1/P2)                               │
│     - 关联到 Milestone                                    │
└─────────────────────────────────────────────────────────┘
                           │
                           ▼
┌─────────────────────────────────────────────────────────┐
│  2. 架构师设计 → 更新技术文档                             │
│     - 读取 Issue 需求                                     │
│     - 设计 API/数据模型                                   │
│     - 更新 docs/ 下的设计文档                              │
└─────────────────────────────────────────────────────────┘
                           │
                           ▼
┌─────────────────────────────────────────────────────────┐
│  3. 开发实现 → 提交代码                                   │
│     - Spawn 子代理写代码                                  │
│     - 遵循代码规范                                        │
│     - Commit + Push                                       │
└─────────────────────────────────────────────────────────┘
                           │
                           ▼
┌─────────────────────────────────────────────────────────┐
│  4. 测试验证 → 更新 Issue 状态                             │
│     - 检查编译通过                                        │
│     - 运行测试（如果有）                                   │
│     - 标记 Issue 为 Ready for Review                       │
└─────────────────────────────────────────────────────────┘
                           │
                           ▼
┌─────────────────────────────────────────────────────────┐
│  5. PM 检查 → 决定是否验收                                │
│     - 所有 P0 完成？→ 请求用户验收                         │
│     - 有阻塞问题？→ 返回步骤 2 或 3                        │
│     - 正常迭代 → 继续下一个 Issue                          │
└─────────────────────────────────────────────────────────┘
                           │
                           ▼
┌─────────────────────────────────────────────────────────┐
│  6. 用户验收 (仅 Phase 结束或 P0 全部完成时)                │
│     - 等待用户确认                                        │
│     - 或继续下一个 Phase                                   │
└─────────────────────────────────────────────────────────┘
```

## 自动化规则

### 何时请求用户介入

| 情况 | 操作 |
|------|------|
| Phase 完成 | 发送验收请求 |
| P0 任务全部完成 | 发送验收请求 |
| 遇到技术决策分歧 | 列出选项，请求拍板 |
| 需求不清晰 | 暂停，请求澄清 |
| 正常迭代中 | **不打扰用户** |

### 何时自动继续

| 情况 | 操作 |
|------|------|
| 当前任务完成 | 自动拉取下一个任务 |
| 测试通过 | 自动标记完成 |
| 编译失败 | 自动修复或记录 Issue |
| 文档过时 | 自动更新 |

## 代码规范

### 包结构

```
com.fizzzli.listapp/
├── ui/                    # UI 层
│   ├── theme/             # 主题
│   ├── components/        # 通用组件
│   ├── screens/           # 页面
│   └── navigation/        # 导航
├── data/                  # 数据层
│   ├── local/             # Room
│   │   ├── dao/
│   │   ├── entity/
│   │   └── ListDatabase.kt
│   └── repository/        # 数据仓库
├── domain/                # 领域层
│   ├── model/             # 领域模型
│   └── usecase/           # 用例
└── di/                    # Hilt 模块
```

### Commit 规范

```
feat: 添加列表创建功能
fix: 修复条目删除崩溃
docs: 更新 README
refactor: 重构数据层
test: 添加单元测试
```

### 分支策略

- `main` - 稳定版本（可发布）
- `develop` - 开发分支（日常合并）
- `feature/xxx` - 功能分支（临时）

## 沟通机制

### 日常迭代
- **不通知用户**，自动推进
- Issue 状态更新在 GitHub

### 需要验收时
- 发送消息给 @Fizzzli
- 包含：完成内容、演示方式、已知问题

### 紧急阻塞
- 立即通知
